TravelMatch AI

A taste-based travel recommendation prototype that goes beyond broad search by modeling traveler preferences, ranking destinations by fit, explaining the reasoning, and organizing transport, stay, food, activities, and reviews around the same travel profile.

Project overview

What the prototype does

Users set what matters most to them in travel, such as views, history, food, safety, nightlife, events, activities, and budget friendliness. The system then scores destinations against that weighted preference profile, explains why a place is a fit, and expands the destination into a full recommendation layer instead of just returning a list of generic search results.

Problem

Most travel search tools are excellent at retrieval, but weaker at expressing personal fit. Two cities can both be “good,” while only one actually matches the type of trip a user wants.

Core solution

TravelMatch AI converts user taste into a weighted profile, ranks destinations by match score, and keeps that same logic consistent across transport, residence, food, activities, and reviews.

Prototype status

The current version is an interactive front-end prototype using curated sample destinations and reviews. It demonstrates the logic, user flow, and visual system even though live APIs and persistent user reviews are not implemented yet.

Differentiation

What makes this different from Google-style travel search

The goal is not to compete on raw search breadth. The goal is to make travel recommendations feel more personal, more explainable, and more decision-oriented.

Typical broad search experience

Search retrieves popular options

Popularity-first discoveryPlaces are often surfaced because they are well known, highly rated, or broadly relevant.
Generic review summariesReviews are usually compressed into a single overall picture rather than interpreted for a specific kind of traveler.
Separate planning layersDestination discovery, hotels, food, activities, and neighborhood tradeoffs can feel like separate searches.
TravelMatch AI approach

The AI predicts fit, not just popularity

Weighted taste profileThe system asks what kind of trip the user wants and uses those weights to score destinations.
Explainable rankingEach destination is accompanied by a match score and explicit reasons such as strong food-history fit or strong scenery-safety balance.
Traveler-type review intelligenceReviews can be filtered by personas such as foodie, solo traveler, budget traveler, or history lover.
Consistent recommendation stackAfter the place is chosen, transport, stay, food, events, and activities all inherit the same taste logic.
Prototype UI walkthrough

Captured interface states

The following visuals document the main screens and interactions in the current prototype. Together, they show the end-to-end logic from preference input to destination selection, recommendation expansion, review interpretation, and feedback-based learning.

1. Taste profile + ranked destinations

The first state establishes the core idea of the product. The user adjusts travel priorities, and the app immediately ranks candidate destinations. Instead of asking the user to type a broad query and manually compare results, the interface translates travel taste into an explainable score. This is the heart of the product concept.

2. Selected destination + full recommendation layer

Once a destination is chosen, the prototype expands into an integrated travel planning layer. Transportation, residence, food, and events are no longer separate searches. They are structured as downstream recommendations based on the same logic that made the destination a fit in the first place.

3. Review intelligence + adaptive learning loop

This section demonstrates the most distinctive AI feature in the current prototype. Reviews are grouped by traveler type so that the user can interpret a place through the lens of a relevant persona. The learning loop then lets users describe what they wanted more or less of in a future trip, which updates the profile for later recommendations.

UX design

How the design evolved

The prototype started as a more ordinary travel search concept and gradually became a preference-matching system. The design evolution below is important because it shows how the project moved away from “AI that searches travel data” and toward “AI that understands travel taste.”

Interaction evolution

1

Keyword-first idea

The earliest concept centered on a user typing interests such as views, history, or food. That version was too close to existing search experiences and did not produce a strong product identity.

2

Weighted preference engine

The next version replaced broad keywords with importance sliders. This created a more legible AI system because destinations could now be ranked by fit instead of simply retrieved by topic similarity.

3

Review intelligence and recommendation stack

The strongest version of the concept added two differentiators: traveler-type review filtering and a full recommendation layer after destination selection. This is the version that became the current prototype.

Core components

Essential parts of the app

1. Preference profiling

The user defines a trip by expressing weighted interest in views, history, food, events, activities, safety, nightlife, and budget. This creates the travel profile that powers the whole app.

2. Destination scoring engine

Each destination carries structured category scores. The app calculates match quality by comparing destination strengths with the user’s weighted interests.

3. Explainable AI summaries

The system explains why a destination surfaced, not just what it is. This makes the recommendation feel less like search output and more like guided judgment.

4. Recommendation stack

After a place is selected, the interface expands into transportation, residence, food, and events so the user can continue planning inside the same product logic.

5. Review intelligence

Reviews are more useful when they come from the right traveler type. The prototype demonstrates this by letting users filter perspectives instead of relying only on overall ratings.

6. Learning loop

The user can describe what they wanted more or less of, and the system updates the preference profile. This begins to turn one-off planning into a personalized travel memory system.

Development plan

How the prototype was built and how it could grow

Area Current prototype Next-stage direction
Front end Single-page web prototype in HTML, CSS, and JavaScript suitable for GitHub Pages deployment. Move to a fuller React or Next.js implementation if the app grows beyond a single-page prototype.
Recommendation logic Weighted scoring using structured destination metadata and category values. Add embedding-based similarity, better ranking heuristics, and user memory across sessions.
AI layer Prototype “AI explanations” are deterministic summaries generated from the scoring logic. Introduce a real LLM layer to produce personalized narratives, tradeoff analysis, and trip comparisons.
Reviews Curated sample reviews tagged by traveler persona. Add a database-backed review system and classify reviews by traveler archetype automatically.
Data sources Static sample destination data used to validate the interaction model. Pull from APIs for destinations, hotels, transport, safety indices, events, and cuisine data.
Training methods No formal model training yet. The prototype focuses on logic, UX, and interaction design. Potential future work includes preference prediction, review clustering, and recommender personalization.
Prompting strategy Prompting was used during design and prototyping to refine product positioning, feature framing, UI copy, and implementation structure. Future prompt chains could generate destination narratives, trip themes, and comparative recommendation explanations.
Vibe coding tools AI-assisted coding support was used to accelerate front-end prototyping and convert the product idea into a working interface quickly. Possible future use of Cursor, Copilot, or other AI IDE tools for larger app development and refactoring.
AI usage note

How AI helped build the prototype

Initial overview

AI was used as a prototyping partner. It helped translate the raw product idea into a clearer structure, identify the strongest differentiator from standard travel search, and scaffold the first working version of the front-end code.

What the human still controlled

The concept direction, feature priorities, product taste, and what counts as success remained human decisions. The prototype reflects deliberate choices about differentiation, UI hierarchy, and the role of review intelligence rather than a blind auto-generated build.

Troubleshooting log

Challenges encountered and current fixes

Challenge 1 · The concept felt too close to normal travel search

The first version focused too much on destination keywords like views, history, or food. That made the app sound useful but not distinct. The solution was to shift from search terms to a weighted taste profile and build the app around fit scoring.

Challenge 2 · The recommendation felt shallow after destination selection

Initially, the app could suggest a destination but did not feel complete afterward. The fix was to add a structured recommendation layer for transportation, residence, food, and activities so the product could continue the planning flow instead of ending at discovery.

Challenge 3 · Reviews were too generic

A simple review list was not enough to differentiate the product. The design was improved by grouping reviews around traveler personas such as foodie, solo traveler, or budget traveler so the user could interpret reviews through a more relevant lens.

Challenge 4 · Scope versus assignment timeline

Features such as live APIs, authentication, and persistent review submission were larger than the available time window. The current fix was to keep the prototype front-end focused and coherent while documenting the clear path for future implementation.
